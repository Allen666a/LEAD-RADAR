# Lead Radar 实用型产品实施方案

版本：P11-P16 执行版  
日期：2026-06-12  
业务范围：海外动态住宅 IP 代理获客  
目标客户：个人、小团队、工作室、小 B 服务团队  

## 1. 核心原则

这套工具不是玩具，也不是“功能越多越好”的后台。

后续所有开发只围绕一个目标：

```text
每天稳定产出少量但高质量、可解释、可跟进、2026 年后的动态住宅 IP 线索。
```

只做有用的、实用的能力。不做鸡肋功能。

判断一个功能要不要做，只看它是否直接提升下面任意一项：

- 有效线索数量
- 线索质量
- 线索时效
- 联系方式补全率
- 跟进效率
- 转化反馈闭环
- 平台采集稳定性
- 垃圾线索过滤能力

如果一个功能只是增加按钮、增加页面、增加概念，但不能让用户更快找到真实客户，就不做。

## 2. 明确不做

以下能力暂不做，除非后续证明能直接带来转化：

- 不做通用爬虫平台。
- 不做大而全 CRM。
- 不做自动群发、自动私信、自动评论、自动关注。
- 不做验证码绕过、登录绕过、风控绕过。
- 不做为了好看而堆复杂 UI。
- 不做没有转化价值的数据大屏。
- 不做无法解释来源和时间的高分线索。
- 不做 2026 年以前的历史线索主推。
- 不做新闻、教程、同行软文、官网文章的获客主线。
- 不做大量平台扩张但没有质量控制的采集。

## 3. 产品定位

Lead Radar 后续按“一套系统，两条获客线”建设。

第一条线：大海捞针  
用于从国内公开内容、社区、问答、论坛、社媒、卖家圈里发现正在表达痛点的人。

第二条线：小 B 线索池  
用于沉淀工作室、小团队、账号服务商、爬虫团队、跨境卖家团队、社媒矩阵团队。

当前阶段重点不是正式 ToB 公司获客，而是个人、小团队、小 B 场景。

## 4. 目标客户优先级

优先级 A：

- TikTok 小店/矩阵/养号团队
- 亚马逊多账号/铺货团队
- Shopee/Lazada/Shein 店群团队
- 账号注册/养号服务商
- 爬虫/数据采集团队
- 指纹浏览器重度用户
- 海外社媒矩阵运营团队

优先级 B：

- Shopify 独立站小团队
- 跨境工具运营者
- 海外广告投放小团队
- 中小型数据采集服务商

低优先级：

- 新手教程读者
- 泛技术学习用户
- 单纯问免费代理的人
- 只讨论 VPN、机场、服务器的人
- 同行代理服务商
- 新闻/资讯/报告内容作者

## 5. 线索硬门槛

一条线索进入主池必须满足：

```text
原文时间 >= 2026 年
来源平台明确
原文链接可打开或可追溯
和动态住宅 IP、账号环境、防关联、爬虫受阻、海外矩阵至少有一个强关联
不是新闻、教程、招聘、官网软文、同行广告
系统能解释为什么值得看
```

没有原文时间的线索：

- 不进入高优先级。
- 默认进入“待人工确认”。
- 不能因为关键词命中就排到前面。

2026 年以前的线索：

- 默认无效。
- 不进入主池。
- 只允许作为历史参考，不作为跟进对象。

## 6. 总体路线

后续执行分成 6 个阶段：

```text
P11 线索质量引擎
P12 国内平台采集稳定化
P13 联系方式瀑布强化
P14 今日作战台
P15 学习闭环
P16 UI 产品化收口
```

每个阶段完成后必须测试，并产出结果报告。

## 7. P11：线索质量引擎

目标：让系统先变准。

### 7.1 要做

- 强化 2026+ 时间门槛。
- 对无时间线索降权或隔离。
- 对教程、新闻、招聘、官网软文、同行广告降权。
- 对动态住宅 IP、住宅代理、轮换 IP、代理池、防关联、多账号、指纹浏览器、403、429、Cloudflare、验证码、封号等强信号加权。
- 按客户类型输出解释：
  - TikTok 矩阵
  - 亚马逊多账号
  - Shopee/Lazada 店群
  - 爬虫/数据采集团队
  - 账号注册/养号服务商
  - 海外社媒矩阵
  - 指纹浏览器用户
- 每条高分线索必须输出：
  - 为什么像目标客户
  - 为什么像动态住宅 IP 需求
  - 原文时间
  - 来源平台
  - 风险点
  - 下一步动作

### 7.2 不做

- 不为了数量扩大低质关键词。
- 不把泛代理、VPN、机场、静态住宅混进高优先级。
- 不把“看起来相关”的教程当客户。

### 7.3 验收标准

- 高优先线索里 2026 年前数据为 0。
- 高优先线索必须显示原文时间。
- 高优先线索至少 70% 能解释出动态住宅 IP 相关理由。
- 教程/新闻/同行软文不再进入高优先。

## 8. P12：国内平台采集稳定化

目标：让采集不是黑盒。

### 8.1 要做

- 每个平台显示采集状态：
  - 可采集
  - 需登录
  - 受阻
  - 待试跑
  - 低质量
  - 暂停
- 后台采集显示进度：
  - 当前平台
  - 当前关键词
  - 已采数量
  - 入库数量
  - 拒绝数量
  - 失败原因
  - 开始时间
  - 更新时间
- 失败原因标准化：
  - 未登录
  - 登录失效
  - 验证码
  - 访问频率过高
  - 页面结构变化
  - 没有新内容
  - 平台拦截
  - 时间不合格
  - 质量不合格
- 对知乎、小红书、抖音、微博等高风控平台使用保守采集策略。
- 采集失败要有可读诊断，不让用户猜。

### 8.2 不做

- 不做验证码破解。
- 不做绕登录。
- 不做高频刷页面。
- 不做点赞、关注、评论、私信。
- 不为了采集量牺牲账号安全。

### 8.3 验收标准

- 用户点击后台采集后能看到进度。
- 每个平台失败时能看到明确原因。
- 采集结果能显示来源平台、原文时间、关键词和拒绝原因。

## 9. P13：联系方式瀑布强化

目标：从“看到需求”变成“能联系”。

### 9.1 要做

联系方式补全按瀑布顺序执行：

```text
原文页
作者主页
平台个人主页
公开网站
about/contact/support/sales 页面
GitHub/Gitee/Profile/博客
同名搜索
人工补充
```

每个联系方式必须保存：

- 类型：微信、QQ、Telegram、邮箱、手机号、网站、表单
- 值
- 来源 URL
- 来源类型
- 置信度
- 发现时间
- 是否需要人工验证

### 9.2 不做

- 不抓取敏感隐私。
- 不从非公开页面取数据。
- 不把搜索结果里的联系方式直接当真，必须回到公开主页验证。
- 不自动触达。

### 9.3 验收标准

- 高价值缺联系方式线索能生成补全路径。
- 每条联系方式能看到来源。
- 补全失败能看到失败原因。
- 联系方式置信度可用于排序。

## 10. P14：今日作战台

目标：用户每天打开工具就知道今天做什么。

### 10.1 要做

首页变成真正的作战台，显示：

- 今天最值得看的新线索
- 今天需要补联系方式的线索
- 今天该跟进的人
- 今天该复访的人
- 哪个平台正在受阻
- 哪些来源质量变差
- 哪些关键词带来高质量线索
- 哪些关键词应该暂停

线索动作统一：

```text
查看证据
补联系方式
加入跟进
标记无效
安排复访
记录反馈
```

### 10.2 不做

- 不做花哨但无行动价值的大屏。
- 不显示一堆无意义总数。
- 不把所有模块堆在首页。

### 10.3 验收标准

- 用户打开首页 30 秒内能知道今天先做哪 3 件事。
- 每个推荐都有原因。
- 每个推荐都能直接进入下一步动作。

## 11. P15：学习闭环

目标：工具越用越准。

### 11.1 要做

记录反馈：

- 有效
- 无效
- 已联系
- 已回复
- 有兴趣
- 已测试
- 成交
- 价格不合适
- 不是动态住宅需求
- 没预算
- 找不到联系方式
- 同行/广告/教程

用反馈更新：

- 来源权重
- 关键词权重
- 客户类型权重
- 平台优先级
- 噪音规则
- 推荐动作

### 11.2 不做

- 不做复杂机器学习黑盒。
- 不做无法解释的自动决策。
- 不因为少量反馈就剧烈改变策略。

### 11.3 验收标准

- 用户标记无效后，同类垃圾线索减少。
- 用户标记成交/测试后，同类线索权重上升。
- 来源效果能按“有效线索/可联系/已回复/成交”排序。

## 12. P16：UI 产品化收口

目标：界面为工作效率服务。

### 12.1 要做

- 继续精简左侧菜单。
- 把重复模块合并。
- 首页只保留作战相关内容。
- 表格减少横向滚动。
- 长内容默认可展开，不遮挡。
- 链接全部可点击。
- 当前模块状态清晰。
- 平台登录、采集、质检、补联系方式入口清晰。
- 详情页成为核心工作页。

### 12.2 不做

- 不为了“看起来高级”牺牲操作效率。
- 不堆无用卡片。
- 不做营销官网式布局。
- 不保留没人用的页面。

### 12.3 验收标准

- 新用户能在 3 分钟内明白主流程。
- 常用动作不超过 2 次点击。
- 页面不再出现遮挡、横向滚动难用、内容截断无法展开等问题。

## 13. 执行顺序

严格按下面顺序推进：

```text
1. P11 线索质量引擎
2. P12 国内平台采集稳定化
3. P13 联系方式瀑布强化
4. P14 今日作战台
5. P15 学习闭环
6. P16 UI 产品化收口
```

如果 P11 没完成，不继续盲目扩平台。  
如果 P12 没完成，不继续增加黑盒采集。  
如果 P13 没完成，不继续堆线索数量。  
如果 P14 没完成，不继续加边缘功能。  
如果 P15 没完成，不继续做不可验证的优化。  
如果 P16 没完成，不继续做无用页面。

## 14. 每轮开发交付格式

每轮开发完成后必须报告：

```text
完成了什么
改了哪些文件
解决了哪个真实问题
哪些能力被明确排除
测试了什么
测试结果
当前还剩什么
下一步建议
```

没有测试，不算完成。  
没有业务价值说明，不算完成。  
不能减少人工判断成本，不算核心功能。

## 15. 最终判断标准

这个工具是否专业，不看功能数量，看这些结果：

- 每天能产出多少条 2026+ 高质量线索。
- 高分线索里垃圾占比是否下降。
- 每条线索是否能解释为什么值得看。
- 是否能找到可触达入口。
- 是否能告诉用户今天该跟谁、怎么跟。
- 用户反馈后，系统是否变得更准。
- 国内平台采集是否可诊断、可暂停、可恢复。

最终目标不是“数据很多”，而是：

```text
少量、高准、可解释、可联系、可跟进、可复盘。
```
