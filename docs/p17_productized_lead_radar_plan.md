# P17 产品化获客工具实施方案

版本：2026-06-13  
业务定位：面向中国市场的海外动态住宅 IP 获客工具，优先服务个人、小团队、工作室，不优先做需要公司资质、发票和正式采购流程的 ToB 大客户。

## 1. 一句话目标

把 Lead Radar 从“采集和筛选后台”升级成“任何人打开就会用的获客助手”。

用户不需要理解采集源、权重、质检、平台矩阵这些内部概念。用户每天只需要回答三个问题：

1. 今天有哪些新客户值得看？
2. 哪些客户现在可以联系？
3. 联系后结果是什么？

## 1.1 执行前审计结论

审计时间：2026-06-13  
当前数据状态：活跃线索 24 条，2026 年以前活跃线索 0 条，未知时间活跃线索 0 条，客户目标 21 个，启用来源 148 个。

结论：

1. 方案方向正确，可以开始执行。
2. 执行顺序必须锁定为 P17 -> P18 -> P19，不能先扩平台、扩来源或做复杂分析。
3. 当前最大短板不是“数据源不够多”，而是“普通用户不知道怎么开始、线索为什么值得联系、联系后怎么推进”。
4. P23 数据源扩展必须延后到 P17-P19 完成后，否则会放大噪音和维护成本。
5. 所有“补联系方式”能力必须限定为公开信息、用户可访问页面和手动验证，不做隐私抓取、不绕登录、不批量骚扰。

执行锁定：

- 先做傻瓜式入口，再做更多来源。
- 先做解释和下一步，再做图表。
- 先做人工私聊助手，再考虑任何自动化触达。
- 先让一个普通人会用，再给高级用户配置项。

## 2. 外部资料和竞品结论

### 2.1 Apollo

Apollo 的核心不是单一数据库，而是把找潜客、补全联系方式、销售触达和 CRM 工作流合在一起。它强调减少人工 prospecting，把用户从重复找人、补资料和写触达中解放出来。

参考：
- https://www.apollo.io/
- https://www.apollo.io/llm-info
- https://www.apollo.io/insights/data-enrichment-tools

对我们的启发：
- 主入口不能叫“平台采集”或“来源管理”，应该叫“找线索”。
- 线索页必须直接告诉用户“为什么这条值得联系”。
- 补联系方式和跟进动作应该在同一条线索上完成，而不是跳很多页面。

### 2.2 Clay

Clay 的强项是数据表式工作流、多个数据源补全、AI research、自动化执行。它把多源数据能力包成一张可操作的表，而不是把每个数据源都暴露给普通用户。

参考：
- https://www.clay.com/
- https://www.understoryagency.com/blog/b2b-data-enrichment-clay-workflows-personalized-outbound-scale

对我们的启发：
- 我们可以保留多平台、多来源、多规则，但默认界面应该是一张“待处理客户表”。
- 高级设置给懂的人用，普通用户只看结果。
- 后续可以做类似“补全瀑布”：主页、公开搜索、社媒、邮箱、Telegram、QQ群关键词逐步补全。

### 2.3 ZoomInfo / Intent Data

ZoomInfo 强调 intent data 的价值不在“有数据”，而在它能驱动优先级、提醒、路由和工作流。2026 年的成熟销售工具会把意图信号和 CRM 记录连接起来，用于自动触发下一步动作。

参考：
- https://pipeline.zoominfo.com/sales/what-is-intent-data-and-how-to-use-it
- https://pipeline.zoominfo.com/sales/buyer-intent-data-tools-sales
- https://pipeline.zoominfo.com/sales/data-enrichment-services-tools

对我们的启发：
- “动态住宅 IP 需求”要拆成意图簇，而不是只堆关键词。
- 线索评分要解释为：客户类型、痛点、时效、可触达性、风险。
- 系统要告诉用户下一步动作，不只是给分数。

### 2.4 lemlist / Sales Engagement

lemlist 和销售节奏类工具强调：找高意向客户、补联系方式、多渠道触达、统一收件箱和可跟踪序列。成熟 cadence 平台会协调邮件、电话、社交触达，并记录每一次互动。

参考：
- https://www.lemlist.com/
- https://help.lemlist.com/en/articles/7939921-enrich-leads-with-finder-verifier-and-phone
- https://pipeline.zoominfo.com/sales/sales-cadence-tools
- https://outreach.ai/resources/blog/sales-cadence

对我们的启发：
- 我们不应该先做复杂自动群发，应该先做手动私聊节奏。
- 需要内置“第一次联系、补充问题、发测试、测试后复访、无回复复访”的话术和提醒。
- 对国内平台尤其要避免批量私信、评论和高频动作，先做人工触达助手。

### 2.5 学术和模型资料：Lead Scoring

lead scoring 的关键是标准化和自动化优先级，传统模型靠人工经验，预测模型利用行为、画像和历史转化数据。研究和行业资料都强调：评分不是最终目的，优先级和资源分配才是目的。

参考：
- https://pmc.ncbi.nlm.nih.gov/articles/PMC11925937/
- https://www.salesforce.com/blog/sales/lead-scoring/
- https://www.madisonlogic.com/blog/lead-scoring-model/

对我们的启发：
- 当前评分规则必须能解释，不能是黑盒数字。
- 用户反馈“成交、无效、没回复、价格不合适、静态 IP 需求”等，要反向调整评分。
- 在数据量不够前，不急着上复杂机器学习；先做规则 + 反馈学习。

### 2.6 GitHub / 开源项目结论

GitHub 上有不少 lead generation、lead scoring、CRM、email finder、OSINT、AI lead generator 项目，但大多数只覆盖一小段能力，比如邮箱猜测、CRM 表、通用爬虫、英文 B2B prospecting。它们可以参考或局部融合，但不能直接替代我们的核心产品。

参考：
- https://github.com/topics/lead-generation
- https://github.com/topics/sales-prospecting
- https://github.com/topics/email-finder
- https://github.com/jivoi/awesome-osint
- https://github.com/brightdata/ai-lead-generator
- https://github.com/ong/awesome-ai-gtm

结论：
- 可以借鉴 CRM pipeline、email finder、OSINT 思路。
- 不建议直接套一个 CRM 或通用 lead generator，因为我们的核心是国内公开平台 + 动态住宅 IP 场景 + 小团队触达。
- 开源能力最适合用于“补联系方式、导出、看板、规则引擎”，不是替代采集策略。

## 3. 我们当前工具审计

### 3.1 已具备能力

1. 线索采集：GitHub、Gitee、V2EX、SegmentFault、知乎会话采集等。
2. 时效过滤：活跃线索已限制为 2026 年及以后，旧线索会归档。
3. 质量判断：可识别痛点、购买意图、教程噪音、同行推广、静态住宅错配等。
4. 客户画像：把线索合并成 prospect，判断客户类型。
5. 补联系方式：已有公开主页和公开搜索方向的补联工作台。
6. 跟进状态：支持新客户、已触达、已发测试、复访、成交、无效等状态。
7. 反馈学习：可以根据结果反馈调整关键词、来源、平台和客户类型权重。
8. UI 主流程已初步收敛：首页、找线索、待联系、跟进中、客户列表、结果反馈。

### 3.2 当前主要问题

1. 找线索还不是傻瓜式入口。
   - 用户不知道应该先选择客户类型、平台还是关键词。
   - 采集设置和结果展示仍然分散。

2. 平台采集状态仍偏后台化。
   - 用户需要理解“已登录、受阻、低质过滤、候选、入库”等概念。
   - 缺少一句话状态和明确修复动作。

3. 线索解释还不够强。
   - 现在有分数和证据，但应该更明确：
     - 为什么像客户？
     - 为什么适合动态住宅 IP？
     - 为什么现在值得联系？
     - 缺什么信息？

4. 补联系方式仍不够产品化。
   - 需要“一键补全”体验，而不是让用户自己跳页面。
   - 需要明确展示找到/没找到/需要人工查的状态。

5. 触达节奏不完整。
   - 缺少针对不同场景的默认话术：
     - TikTok 矩阵
     - 亚马逊多账号
     - 爬虫/数据采集
     - 社媒矩阵
     - Shopify/独立站
   - 缺少 1 天、3 天、7 天复访模板。

6. 结果反馈的闭环还不够强制。
   - 用户如果不反馈，系统学习会慢。
   - 需要在处理动作后提醒记录结果。

7. 国内平台稳定性仍是最大变量。
   - 知乎、贴吧、小红书、抖音、微博等有风控和页面变化。
   - 当前策略必须保持低频、人工登录、只读公开内容，不做批量互动。

## 4. 产品路线原则

### 4.1 只做有用能力

做：
- 找到 2026 年的新需求。
- 判断是否是动态住宅 IP 的目标客户。
- 找到能联系的入口。
- 给出能直接私聊的话术。
- 记录结果并优化下一批线索。

不做：
- 花哨图表。
- 大而全 CRM。
- 自动批量私信。
- 绕验证码、绕登录、规避平台风控。
- 只增加按钮但不能提高线索质量的功能。

### 4.2 默认给普通用户，隐藏高级能力

普通用户只看：
- 首页
- 找线索
- 待联系
- 跟进中
- 客户列表
- 结果反馈

高级用户才看：
- 平台登录
- 平台状态
- 搜索词
- 采集源
- 线索质检
- 效果分析

### 4.3 线索宁少勿滥

错误线索会浪费人的时间。工具的核心指标不是采集数量，而是：
- 新鲜度
- 相关度
- 可触达性
- 跟进转化率

## 5. 新信息架构

### 5.1 首页：今天该联系谁

目标：打开即知道今天做什么。

必须展示：
- 今天建议联系数量。
- 高优先级数量。
- 到期复访数量。
- 反馈覆盖率。
- 先做 3 件事。
- 待联系客户。
- 需要处理的平台。

不展示：
- 采集源细节。
- 原始候选数量。
- 复杂模型字段。

### 5.2 找线索

目标：傻瓜式启动采集和查看新线索。

页面结构：
1. 你想找哪类客户？
   - TikTok 矩阵/小店
   - 亚马逊多账号/铺货
   - Shopee/Lazada/Shein 店群
   - 爬虫/数据采集
   - 社媒矩阵
   - Shopify/独立站
   - 账号注册/养号服务商

2. 想从哪里找？
   - 国内平台优先
   - 技术社区
   - 跨境卖家社区
   - 已登录平台
   - 全部

3. 开始找线索
   - 按钮：开始找
   - 状态：正在找、找到多少、过滤多少、可联系多少。

4. 结果列表
   - 标题
   - 来源平台
   - 原文时间
   - 为什么值得看
   - 下一步
   - 原文链接

验收标准：
- 新用户 30 秒内知道如何找线索。
- 找线索页面不出现“source、keyword weight、quality gate”等内部词。

### 5.3 待联系

目标：只处理可触达或值得补联的客户。

页面结构：
- 已有联系方式
- 缺联系方式
- 需要人工确认
- 一键补全公开联系方式
- 复制话术
- 标记已联系

验收标准：
- 每条客户必须有“联系动作”或“缺什么信息”。
- 没联系方式的客户不能混在可联系客户里。

### 5.4 跟进中

目标：防止联系后丢失。

状态：
- 已触达
- 已发测试
- 1 天后复访
- 3 天后复访
- 7 天后复访
- 已成交
- 无效

验收标准：
- 每个跟进客户都有下一次动作时间。
- 首页能显示到期复访。

### 5.5 结果反馈

目标：让系统变聪明。

反馈类型：
- 成交
- 有意向
- 无回复
- 价格不合适
- 不是动态住宅 IP
- 只要静态住宅
- 同行/推广
- 教程/学习
- 联系方式无效

验收标准：
- 每个处理动作后都能快速记录结果。
- 反馈会影响关键词、来源、平台和客户类型权重。

## 6. 数据和评分模型升级

### 6.1 意图簇

不再只看关键词，改成意图簇：

1. 防关联/账号环境
   - 多账号、店群、矩阵、防关联、环境异常、封号、账号关联。

2. 平台业务场景
   - TikTok 小店、亚马逊多账号、Shopee、Shopify、FB/IG/YouTube、跨境独立站。

3. 爬虫/数据采集痛点
   - 403、429、Cloudflare、验证码、IP 被封、请求太频繁。

4. 代理类型匹配
   - 动态住宅、轮换住宅、海外住宅、HTTP/SOCKS5、国家地区、并发、流量。

5. 排除项
   - 静态住宅、免费代理、教程、测评、官网推广、同行广告、历史旧帖。

### 6.2 新评分公式

建议评分：

总分 = 场景匹配 30 + 痛点强度 25 + 动态住宅匹配 20 + 时效 15 + 可触达 10 - 风险扣分

风险扣分：
- 旧线索：直接归档。
- 同行推广：直接归档。
- 静态住宅需求：降权或归档。
- 教程/文章：归档。
- 无法判断时间：默认不进入主线索池。

### 6.3 解释字段

每条线索必须展示：
- 客户类型
- 需求场景
- 痛点
- 动态住宅匹配原因
- 原文时间
- 缺失信息
- 建议动作

## 7. 平台采集策略

### 7.1 国内公开平台

优先级：
1. V2EX：适合技术、爬虫、代理异常、小团队。
2. SegmentFault：只保留问答，不要文章/标签页。
3. 知乎：适合搜索公开问答，但质量波动大，要强过滤。
4. 百度贴吧：可能有个人/小团队，但噪音高。
5. 小红书/抖音/微博/B站：适合人工登录后低频公开搜索，优先观察，不做批量互动。

### 7.2 跨境平台

优先级：
1. WeAreSellers / 卖家论坛
2. Amazon Seller Forum 中文讨论
3. 跨境相关公开论坛

### 7.3 技术平台

优先级：
1. V2EX
2. GitHub Issues
3. Gitee Issues
4. SegmentFault 问答

说明：
- GitHub/Gitee 对爬虫/代理需求有效，但对 TikTok/跨境卖家不一定精准。
- 技术平台更适合“爬虫/数据采集团队”这一客户类型。

## 8. 合规和风控边界

必须坚持：
- 只读公开页面。
- 用户自己登录，工具不绕过登录。
- 不破解验证码。
- 不批量评论、关注、私信。
- 不高频访问。
- 对受阻平台暂停并提示人工处理。

原因：
- 国内平台风控强。
- 账号价值高，不能为了多采几条线索冒封号风险。
- 我们的产品定位是获客助手，不是反风控工具。

## 9. 实施路线图

### P17：产品化信息架构和找线索入口

目标：让任何人知道怎么开始。

任务：
1. 重做找线索页面。
2. 增加客户类型选择。
3. 增加平台选择。
4. 增加“开始找线索”按钮。
5. 展示采集进度。
6. 结果列表展示“为什么值得看”。

范围锁定：
- 只改“找线索”主入口和结果呈现，不重写全部采集引擎。
- 默认推荐客户类型，不要求用户先懂关键词。
- 默认国内优先，但允许选择技术社区、跨境社区、全部平台。

不做：
- 不做复杂数据图表。
- 不做批量私信。
- 不暴露 source、quality gate、keyword weight 等内部字段。

验收：
- 新用户不看文档也能开始找线索。
- 页面不出现内部字段。
- 用户点击 1 个按钮即可开始找线索或看到为什么暂时不能找。
- 找线索结果必须显示来源平台、原文时间、线索理由、下一步动作。

风险：
- 如果直接复用旧页面，仍然会像后台。
- 如果入口放太多筛选项，新手会不知道怎么选。

### P18：线索解释器

目标：每条线索都能解释清楚。

任务：
1. 增加线索解释字段。
2. 详情页重排：时间、来源、客户类型、痛点、匹配原因、缺失信息、下一步。
3. 列表页显示一句话理由。
4. 对低质量线索展示归档原因。

范围锁定：
- 先用规则解释，不急着上大模型黑盒评分。
- 解释必须围绕动态住宅 IP，不解释泛代理、静态住宅、大 ToB 采购。

不做：
- 不写很长的 AI 总结。
- 不把旧评分字段原样堆到页面。
- 不让用户猜“分数为什么高”。

验收：
- 用户看一条线索 10 秒内知道值不值得联系。
- 每条主池线索必须有：客户类型、需求场景、痛点、时效、动态住宅匹配原因、缺失信息、下一步。
- 静态住宅、教程、同行推广必须有明确归档原因。

风险：
- 如果解释太长，会降低处理效率。
- 如果解释不区分动态/静态住宅，会继续误导跟进。

### P19：补联系方式工作流

目标：让线索可联系。

任务：
1. 一键补全公开联系方式。
2. 展示微信/QQ/TG/邮箱/主页/群入口。
3. 标记来源和可信度。
4. 找不到时给人工搜索建议。

范围锁定：
- 只查公开信息、原文主页、公开搜索线索。
- 每个联系方式必须标记来源和可信度。
- 找不到也要给人工下一步，而不是沉默失败。

不做：
- 不抓取私信、通讯录、隐藏联系方式。
- 不绕登录、不绕验证码。
- 不保证一定能找到联系方式。

验收：
- 每条待联系客户都有联系方式状态。
- 可联系和不可联系分开。
- 可联系客户必须能一键复制联系方式或打开来源。
- 不可联系客户必须显示缺什么、建议怎么查。

风险：
- 公开联系方式少，自动补全率可能不高。
- 如果不标可信度，容易把错误联系方式当真。

### P20：触达话术和跟进节奏

目标：减少人工想话术。

任务：
1. 按客户类型生成首聊话术。
2. 生成 1 天、3 天、7 天复访话术。
3. 生成发测试包话术。
4. 生成测试后回访问题。
5. 一键复制并记录动作。

范围锁定：
- 只做人工复制话术和提醒，不做自动群发。
- 话术必须按客户类型变化。
- 默认语气要像真人私聊，不像广告。

不做：
- 不做自动私信。
- 不做批量评论。
- 不做骚扰式高频 follow-up。

验收：
- 用户点开客户即可复制话术。
- 每次动作后能安排下一步。
- 话术必须覆盖 TikTok、亚马逊、爬虫、社媒矩阵、Shopify/独立站至少 5 类。
- 记录“已触达/已发测试/无回复/成交/无效”后状态正确流转。

风险：
- 话术太模板化会降低转化。
- 如果没有结果反馈，节奏会变成空转。

### P21：反馈学习强化

目标：越用越准。

任务：
1. 处理动作后强提示记录结果。
2. 反馈类型标准化。
3. 自动调整关键词、来源、客户类型权重。
4. 首页展示“因为反馈，系统做了什么调整”。

范围锁定：
- 先强化规则学习，不急着引入复杂机器学习模型。
- 反馈必须直接影响：关键词、来源、平台、客户类型。

不做：
- 不做无法解释的模型分。
- 不做大屏式分析报表。

验收：
- 反馈后下一批线索排序发生合理变化。
- 成交/无效/静态错配/同行推广等反馈必须可追踪到调整记录。
- 首页显示最近一次学习动作。

风险：
- 数据量少时，自动学习可能过拟合。
- 需要保留人工恢复权重的能力。

### P22：平台健康和采集稳定性

目标：用户知道为什么采不到。

任务：
1. 平台状态卡简化成：可用、需登录、受阻、低质过滤、暂停。
2. 每个平台给一句话原因。
3. 每个平台给一个修复按钮。
4. 采集过程展示进度。
5. 高风险平台自动降频。

范围锁定：
- 重点解释状态，不追求突破平台风控。
- 平台状态必须给出普通人能懂的文案。

不做：
- 不破解验证码。
- 不绕登录。
- 不高频重试。
- 不做账号风控规避。

验收：
- 用户不会再问“到底有没有在采集”。
- 平台受阻时能看到明确原因。
- 采集按钮点击后必须显示进度、候选数、入库数、过滤原因或失败原因。
- 平台受阻必须给出一个安全动作，例如“重新检查登录”“暂停该平台”“降低频率”。

风险：
- 国内平台页面变化快，需要持续维护。
- 状态太技术化会让用户仍然看不懂。

### P23：高质量数据源扩展

目标：扩大线索池，但不牺牲质量。

任务：
1. 继续优化 V2EX。
2. 只保留 SegmentFault 问答页，不保留文章/标签。
3. 增加跨境卖家社区源。
4. 增加国内平台会话搜索模板。
5. 每个来源都有质量分和暂停策略。

范围锁定：
- 只能在 P17-P19 完成后启动。
- 每新增一个来源，必须同时有时效解析、噪音过滤、质量验收。
- 优先新增能带来个人/小团队/工作室线索的平台。

不做：
- 不为了数量接入新闻、官网文章、同行营销站。
- 不接入无法判断时间的来源。
- 不接入只能产生教程噪音的来源。

验收：
- 每周新增可跟线索数量提升。
- 旧线索、教程、同行推广不进入主池。
- 新来源上线前必须跑小批量试采，人工审计至少 20 条候选。
- 新来源连续低产要自动降频或暂停。

风险：
- 扩源最容易把工具重新变乱。
- 来源越多，维护成本越高。

## 10. 不做清单

短期不做：
- 自动批量私信。
- 自动评论引流。
- 破解验证码。
- 自动养号。
- 大公司 ToB CRM。
- 发票、合同、企业采购流程。
- 花哨图表和无用仪表盘。

## 10.1 延后或砍掉的能力

审计后明确延后：

1. 大规模扩平台。
   - 原因：当前还没有把找线索入口、解释、补联系方式做顺，扩平台只会放大噪音。

2. 大模型自动销售助手。
   - 原因：现阶段需要可解释规则和真人私聊，不需要黑盒 AI 销售。

3. 全自动触达。
   - 原因：国内平台风险高，而且用户当前更需要安全的人工触达助手。

4. 大屏分析和复杂报表。
   - 原因：当前样本量小，最重要的是今天联系谁，不是看复杂图表。

5. 企业级 CRM。
   - 原因：当前业务不优先做需要公司资质、发票、合同的大 ToB。

6. 新闻/官网/同行营销内容采集。
   - 原因：这类内容不是客户需求，容易污染线索池。

7. 不能解析时间的来源。
   - 原因：当前硬要求是 2026 年及以后的有效线索，时间不明默认不进主池。

保留但降级为高级设置：

1. 来源管理。
2. 搜索词权重。
3. 平台矩阵。
4. 线索质检。
5. 效果分析。

这些能力对系统有用，但不应该暴露在普通用户主流程里。

## 11. 成功指标

第一阶段指标：
- 新用户 1 分钟内知道如何开始。
- 活跃线索 100% 有原文时间。
- 2026 年以前线索 0 条进入主池。
- 每条线索都有来源、时间、理由、下一步。
- 待联系客户中 80% 有明确联系方式状态。

第二阶段指标：
- 每天稳定产出 5-20 条可人工判断线索。
- 每周至少沉淀 10 条可联系客户。
- 反馈覆盖率超过 60%。
- 无效线索比例持续下降。

第三阶段指标：
- 能区分不同客户类型的转化率。
- 能根据反馈自动推荐关键词和平台。
- 能形成稳定的私域触达节奏。

## 12. 推荐执行顺序

最优先：
1. P17 找线索入口。
2. P18 线索解释器。
3. P19 补联系方式。

第二优先：
4. P20 触达话术和节奏。
5. P21 反馈学习强化。

第三优先：
6. P22 平台健康。
7. P23 数据源扩展。

原因：
- 先解决“用户不会用”和“线索看不懂”。
- 再解决“联系不上”和“不会说”。
- 最后扩大来源，否则只会放大噪音。

## 12.1 阶段开工和完成标准

每个 P 阶段开工前必须回答：

1. 这个阶段能不能让用户更容易找到客户、联系客户或成交？
2. 是否会增加普通用户理解成本？
3. 是否符合动态住宅 IP、小团队/个人/工作室定位？
4. 是否会引入平台账号风险？
5. 是否能在页面上被用户直接感知？

如果 1 是否，或者 2/4 是，则暂缓。

每个 P 阶段完成后必须验证：

1. 页面可打开，核心路径返回 200。
2. 新用户路径不超过 3 步。
3. 不出现旧线索和未知时间线索进入主池。
4. 关键按钮有结果反馈，不让用户猜是否执行。
5. 至少用真实数据截图检查一次 UI 是否拥挤、遮挡、横向溢出。
6. 写一份阶段报告，说明做了什么、没做什么、剩余风险。

## 13. 最终产品形态

理想状态下，用户每天只做 5 个动作：

1. 打开首页。
2. 看今天建议联系谁。
3. 点开客户，看为什么值得联系。
4. 复制话术去私聊。
5. 回来记录结果。

系统后台自动做：
- 找新线索。
- 过滤旧线索。
- 判断客户类型。
- 补联系方式。
- 推荐下一步。
- 根据反馈优化。

这就是我们要做的“傻瓜式获客工具”。
