# Lead Radar 双模式获客系统实施方案

## 1. 背景和目标

Lead Radar 的业务目标不是做通用爬虫工具，而是做一个面向“海外动态住宅 IP 代理业务”的专业获客系统。

核心客户诉求集中在四类：

- 多账号防关联
- 养号、矩阵、店群运营
- 爬虫、数据采集、价格监控
- 跨国家、跨平台、长期消耗型代理 IP 需求

因此系统不能只做“采集帖子”，而要逐步升级为：

```text
发现需求 -> 判断客户价值 -> 补全客户画像 -> 找到联系方式 -> 销售跟进 -> 转化反馈
```

本方案确定产品主架构为“一套系统，两条获客流水线”。

## 2. 产品主架构

系统分为两种获客模式：

```text
Lead Radar
├─ 需求雷达：大海捞针模式
│  ├─ 社区、问答、论坛、社媒、评论、GitHub issue
│  ├─ 找正在喊痛的人和团队
│  └─ 产出痛点线索、即时需求、话题趋势
│
├─ B2B 客户库：企业客户模式
│  ├─ 公司官网、招聘、GitHub 组织、目录、工具站、跨境服务商
│  ├─ 找有长期代理 IP 消耗能力的公司
│  └─ 产出公司画像、意图信号、联系方式、销售路径
│
└─ 统一中台
   ├─ 去重合并
   ├─ ICP 评分
   ├─ 联系方式补全
   ├─ 跟进池
   ├─ 销售节奏
   └─ 转化反馈
```

不做两个独立系统。原因是数据入库、评分、联系方式、跟进状态、CRM 流程都应该共用；但两种模式的采集策略、页面视角、评分权重必须分开。

## 3. 模式一：需求雷达

### 3.1 定位

需求雷达负责“大海捞针”，目标是发现正在表达痛点的人、工作室、小团队、开发者和运营人员。

它回答的问题是：

```text
今天哪里有人正在为代理 IP、防关联、封号、403、爬虫受阻而烦？
```

### 3.2 适合来源

优先来源：

- 知乎
- 百度贴吧
- 小红书
- 抖音
- B 站
- 微博
- V2EX
- SegmentFault
- LearnKu
- CSDN
- OSChina
- GitHub issues / discussions
- 卖家论坛
- 跨境社区
- 私域导入

### 3.3 核心信号

高相关信号：

- TikTok 多账号防关联
- TikTok 小店矩阵
- 亚马逊多账号
- Shopee / Lazada / Shein 店群
- 指纹浏览器代理配置
- IP 被封
- 爬虫 403 / 429
- Cloudflare 验证
- 动态住宅 IP
- 轮换住宅代理
- 代理池不稳定
- 海外社媒矩阵
- 注册、养号、过验证

低质量或应剔除信号：

- 新手教程
- 泛泛技术文章
- 新闻资讯
- 同行官网软文
- 静态住宅 IP 需求
- 免费代理、机场、翻墙
- 只讨论普通 VPS 或云服务器

### 3.4 产出

需求雷达输出的是“痛点线索”，字段重点：

```text
标题
平台
作者/账号
原文链接
痛点类型
匹配关键词
需求强度
风险等级
是否可触达
建议动作
```

### 3.5 评分逻辑

需求雷达评分重点：

- 痛点是否明确
- 是否和动态住宅 IP 直接相关
- 是否出现当前阻塞场景
- 是否可联系
- 是否存在合规风险
- 是否是同行/广告/教程噪音

示例：

```text
强痛点 +30
动态住宅 IP 明确需求 +25
防关联/多账号/爬虫受阻 +20
有联系方式 +15
新手教程 -25
同行广告 -40
新闻/资讯 -50
```

## 4. 模式二：B2B 客户库

### 4.1 定位

B2B 客户库负责找真正有预算、长期复购、消耗量大的企业客户。

它回答的问题是：

```text
哪些公司或团队天然需要长期使用动态住宅 IP？
```

### 4.2 适合来源

优先来源：

- 公司官网
- 招聘网站
- GitHub 组织和项目
- SaaS 产品目录
- 跨境服务商目录
- 电商工具目录
- 广告验证工具公司
- 价格监控公司
- SERP / SEO 工具公司
- 数据采集公司
- AI 数据集公司
- 舆情监控公司
- 出海营销自动化公司

### 4.3 目标客户类型

高价值客户：

- 数据采集团队
- 价格监控公司
- 广告验证公司
- SERP API / SEO 工具
- 电商情报公司
- 舆情监控公司
- 金融数据公司
- AI 数据集公司
- TikTok / Shopee / Amazon 多账号服务商
- 出海 SaaS
- 海外社媒营销自动化团队
- 账号注册 / 养号服务商

低优先级客户：

- 单店铺卖家
- 纯内容培训号
- 免费工具站
- 个人教程作者
- 同行代理服务商
- 无法判断业务场景的普通公司

### 4.4 B2B 意图信号

强信号：

- 招聘爬虫工程师
- 招聘数据采集工程师
- 招聘反爬、风控、账号安全相关岗位
- 官网出现 scraping / crawler / proxy / anti-bot / data extraction
- GitHub 有 scraper / crawler / spider / proxy pool 项目
- 产品是 price monitoring / SERP API / ad verification / e-commerce intelligence
- 面向多个国家和地区提供数据服务
- 明确做跨境店群、账号矩阵、社媒自动化

中等信号：

- 使用 Shopify / Amazon / TikTok / Shopee 生态
- 招聘增长、投放、海外运营岗位
- 官网有 API、数据接口、自动化能力
- 多语言、多国家站点

弱信号：

- 只是普通跨境卖家
- 只有泛泛出海服务描述
- 没有技术或规模化需求证据

### 4.5 产出

B2B 客户库输出的是“公司/团队画像”，字段重点：

```text
公司名
官网
国家/地区
行业
客户类型
业务场景
意图信号
代理需求理由
技术栈/产品线索
招聘信号
GitHub 信号
联系方式
联系人角色
ICP 分数
成交概率
跟进状态
下一步动作
```

### 4.6 评分逻辑

B2B 评分重点：

- 是否公司级客户
- 是否有长期代理消耗场景
- 是否有明确意图信号
- 是否能找到商务联系方式
- 是否有预算可能性
- 是否和动态住宅 IP 强匹配

示例：

```text
公司级实体 +20
数据采集/价格监控/广告验证业务 +30
招聘爬虫/数据采集岗位 +25
官网出现 proxy/scraping/crawler +20
GitHub 有相关项目 +15
有 sales/contact/business 邮箱 +15
只有普通卖家信号 -15
同行代理服务商 -40
```

## 5. 统一中台

两条流水线最终进入同一个中台，避免重复建设。

### 5.1 去重和合并

需要支持：

- 同一账号多条线索合并
- 同一公司多来源合并
- 同一官网、域名、GitHub 组织合并
- 同一联系方式合并
- 社区线索转 B2B 公司画像

合并优先级：

```text
域名 > 公司名 > GitHub 组织 > 主页 URL > 联系方式 > 平台账号
```

### 5.2 ICP 评分

统一 ICP 评分要分两套权重：

- 需求雷达 ICP：痛点强度优先
- B2B ICP：公司价值和长期消耗优先

最终输出统一字段：

```text
ICP 分数
客户类型
产品匹配度
需求理由
风险原因
推荐动作
```

### 5.3 联系方式补全

联系方式来源分层：

1. 官网 contact / about / pricing / support / sales
2. 公司邮箱格式
3. GitHub profile / org
4. LinkedIn / X / Telegram / Discord / YouTube
5. 国内平台主页
6. 人工补充

系统原则：

- 优先公司级商务联系方式
- 不采集敏感个人数据
- 不做绕登录、验证码、付费墙
- 不做批量骚扰

### 5.4 跟进和 CRM

统一状态：

```text
new
qualified
need_contact
contact_enrich
contacted
interested
trial_sent
negotiating
won
lost
invalid
competitor
compliance_review
```

每个客户应记录：

- 最近一次触达时间
- 下一次跟进时间
- 触达渠道
- 回复结果
- 失败原因
- 成交概率
- 备注

## 7. 数据源实施顺序

### 阶段一：先稳住需求雷达

目标：让大海捞针模式更准，不继续灌入教程、新闻、同行软文。

任务：

- 清理关键词策略
- 重新区分强痛点词、弱痛点词、噪音词
- 强化来源质检
- 对国内平台结果单独打“痛点强度”
- 需求雷达页面独立出来

验收：

- 首页不再混合展示所有噪音
- 每条线索明确显示来源平台
- 新手教程、新闻、同行文章明显减少
- 高分线索大多和动态住宅 IP 场景有关

### 阶段二：建设 B2B 客户库

目标：从帖子视角升级到公司视角。

任务：

- 新增 B2B 客户库页面
- 新增公司画像服务
- 支持从官网、域名、GitHub 组织合并客户
- 新增 B2B 评分模型
- 新增 B2B CSV 导出

验收：

- 可以看到公司级客户列表
- 每家公司有代理需求理由
- 每家公司能看到意图信号来源
- 能按 ICP 分数、客户类型、是否有联系方式筛选

### 阶段三：加入 B2B 信号源

目标：找到更高价值客户，而不是只靠社区求助。

优先源：

- GitHub 组织/项目
- 公司官网关键词扫描
- 招聘关键词源
- SaaS/工具目录
- 跨境服务商目录

验收：

- 每天能新增公司级候选客户
- B2B 客户库中不是只有论坛账号
- 能看到招聘、官网、GitHub、目录类信号

### 阶段四：联系方式和 CRM 闭环

目标：让系统真正能推进销售。

任务：

- 强化公开商务联系方式补全
- 增加联系方式可信度
- 增加触达记录
- 增加回复结果
- 增加成交概率
- 增加来源 ROI 分析

验收：

- 销售能从系统直接知道今天联系谁
- 每个客户有下一步动作
- 能统计哪个来源带来可触达客户
- 能统计哪个来源带来成交或有效回复

### 阶段五：Agent 自动化

目标：让后台自动跑关键任务，但不做黑盒乱采集。

Agent 类型：

- source_auditor：来源质量审计
- demand_radar_collector：需求雷达采集
- b2b_signal_collector：B2B 信号采集
- company_enricher：公司画像补全
- contact_enricher：联系方式补全
- icp_judge：ICP 评分
- crm_scheduler：跟进节奏生成

验收：

- 任务可追踪
- 失败可重试
- 每个 Agent 有日志
- 不会因为一个平台受阻导致全系统卡住

## 9. 合规边界

系统必须坚持：

- 优先公司级数据
- 优先公开商务联系方式
- 不绕验证码
- 不绕登录限制
- 不批量私信
- 不批量骚扰
- 不采集敏感个人数据
- 不做账号风控规避工具
- 触达要保留人工确认

对于国内平台：

- 登录态采集只读取公开搜索结果
- 不点赞、不关注、不评论、不私信
- 失败和受阻要暂停
- 高风险平台优先人工导入或低频采集

## 11. 验收标准

这个系统是否变得专业，不看按钮多不多，看以下指标：

```text
每天新增多少高相关线索
每天新增多少公司级客户
高分线索中噪音比例是否下降
多少客户能补到联系方式
多少客户进入跟进池
多少客户有明确下一步动作
哪些来源真实带来有效客户
销售是否能用这个系统完成一天工作
```

阶段性目标：

- 需求雷达：高分线索中 70% 以上和动态住宅 IP 相关
- B2B 客户库：每批至少产出 10 个有明确代理需求理由的公司
- 补联系方式：高价值客户 30% 以上能找到公开商务触达入口
- 跟进池：每个 A 类客户都有下一步动作
- 作战台：每天能告诉用户今天该跑什么、该修什么、该跟谁

## 12. 总结

Lead Radar 后续路线不是继续盲目扩平台，而是升级成双模式系统：

```text
需求雷达负责发现即时痛点。
B2B 客户库负责沉淀高价值企业客户。
统一中台负责评分、补全、跟进和转化。
```

这条路线最符合海外动态住宅 IP 代理业务。因为真正有价值的客户不是所有“提到代理”的人，而是正在承受防关联、爬虫受阻、账号矩阵、价格监控、广告验证等痛点，并且有持续代理 IP 消耗能力的团队或公司。

## 13. 外部资料审计后的修订原则

本方案已根据成熟获客、销售情报、CRM、数据补全、意图数据和合规资料做过一次外部审计。详细审计见：

```text
docs/dual_mode_acquisition_audit.md
```

审计后确认：

- 双模式方向正确。
- “一套系统，两条流水线”正确。
- 下一步不应继续盲目扩源，而应先强化对象模型、评分、补全和反馈闭环。

### 13.1 数据对象分层

后续实现必须明确区分：

```text
Signal -> Prospect -> Company/Account -> Contact -> Activity
```

含义：

- Signal：帖子、搜索结果、招聘、官网关键词、GitHub 项目、目录条目等原始信号。
- Prospect：可被跟进的线索对象，可能来自个人、账号、团队或公司。
- Company/Account：公司级客户主对象，是 B2B 客户库的核心。
- Contact：联系方式和联系人入口。
- Activity：触达、跟进、回复、复查、成交或无效记录。

需求雷达是 Signal 池，不是最终客户库。

B2B 客户库是 Company/Account 池，不是帖子列表。

### 13.2 评分体系拆分

后续不能只依赖一个 `lead_score`。评分需要拆成：

```text
fit_score：客户类型是否适合动态住宅 IP 业务
intent_score：最近是否出现明确需求或行为信号
contact_score：是否可触达，联系方式是否可信
risk_score：合规、噪音、同行、骚扰或错误触达风险
```

最终优先级可以由这四个分数合成，但页面和数据表中必须能解释每个分数的原因。

### 13.3 B2B 以公司为主对象

B2B 客户库必须以 Company/Account 为主对象，合并优先级为：

```text
domain > normalized_company_name + country > github_org > official_social_profile > business_email_domain
```

B2B 页面第一列应该是公司和官网，而不是帖子标题。

### 13.4 补全必须有来源和置信度

联系方式和公司画像补全不能只保存结果，还要保存：

```text
value
source_url
source_type
confidence
fetched_at
failure_reason
```

补全分三类：

- company_enrichment：公司名、域名、行业、国家、规模、产品。
- signal_enrichment：招聘、官网关键词、GitHub、目录、社媒。
- contact_enrichment：邮箱、表单、LinkedIn、Telegram、微信、商务入口。

### 13.5 触达保持人工确认

系统可以生成触达建议和销售序列草稿，但不应做自动群发。

最小规则：

- 没有明确联系方式，不进入触达。
- 风险高，不进入触达。
- 同行、广告、教程，不进入触达。
- 发送前必须人工确认。
- 有回复、退订、无效、投诉，停止序列。

### 13.6 合规和抑制名单

后续必须补充：

```text
source_permission_status
personal_data_flag
lawful_basis_note
retention_until
do_not_contact
suppression_reason
```

对于国内平台，继续坚持：

- 只读取公开结果。
- 不点赞、不关注、不评论、不私信。
- 不绕验证码和登录限制。
- 受阻就暂停或转人工。
