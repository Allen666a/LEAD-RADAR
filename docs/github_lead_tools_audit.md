# GitHub 获客工具源码审计

审计时间：2026-06-14

目标：下载并研究一批开源获客、销售线索、OSINT、联系人发现工具，判断哪些能力值得融合到 Lead Radar。结论只服务一个方向：海外动态住宅 IP 的个人、小团队、工作室获客，不做花哨功能。

## 2. 关键发现

### OpenOutreach 最像产品

它不是简单爬列表，而是完整链路：

- Onboarding：先配置产品、目标客户、活动目标。
- Search：生成搜索词，找候选人。
- Qualification：用 LLM 判断是否匹配。
- ML：用 embedding + Gaussian Process 做主动学习和排序。
- CRM：Lead / Deal / Outcome / Task 状态清晰。
- Queue：任务定时、重试、失败诊断、跟进节奏。
- Contact：连接后再抓联系信息，跟进消息进入记录。

不能直接照搬的地方：

- 它重 LinkedIn 自动化，账号和 ToS 风险更高。
- 它依赖 LLM/API，和我们当前“低依赖、可先本地跑”的路线不完全一致。
- 它更偏英文 LinkedIn 销售，不是国内平台和动态住宅 IP 场景。

可以迁移的地方：

- 状态机：候选 -> 待质检 -> 待补联系方式 -> 待跟进 -> 已触达 -> 成交/无效。
- 任务队列：每个来源独立任务、进度、失败原因。
- 线索解释：为什么值得看、缺什么、下一步做什么。
- 主动学习思想：先不要上复杂 ML，但要保留“人工反馈影响规则权重”的数据结构。

### Bright Data 项目验证了一个主流方向

它的核心不是“更多按钮”，而是：

用户用自然语言说要找谁，系统解析成结构化条件，调用数据源，再用 AI 打分和给触达建议。

对我们的启发：

- 未来可以做一个极简入口：“我想找 TikTok 矩阵/亚马逊多账号/爬虫团队”，系统自动选择来源和关键词。
- 不应该让用户理解一堆平台、来源、工具箱。
- 但它依赖 Bright Data 和 OpenAI，不能作为我们当前免费/低成本主线。

### Google Maps 工具不适合做主线

这些项目适合找餐厅、诊所、本地商家，对我们的客户画像不精准。

可借：

- 导出能力。
- 采集进度。
- 数据字段完整度评分。
- 本地浏览器采集的 UI 反馈。

不建议把 Google Maps 当主来源，因为动态住宅 IP 的需求更多出现在：

- 跨境电商卖家社区。
- 技术社区和爬虫社区。
- 指纹浏览器/账号矩阵相关讨论。
- 私域/论坛/问答/开源 Issue。
- 平台风控、封号、验证、403、429、Cloudflare、账号关联等痛点场景。

### QueryTool 的价值是“搜索模板矩阵”

它本质是 OSINT 查询模板库。我们应该把这个思想迁移成 Lead Radar 的“动态住宅 IP 查询矩阵”：

- 业务场景词：TikTok 小店、亚马逊多账号、Shopee 店群、Shopify 多店、社媒矩阵、爬虫 403。
- 痛点词：防关联、封号、环境异常、验证码、Cloudflare、IP 被封、代理不稳定。
- 购买意向词：求推荐、哪里买、稳定、便宜、测试、替代、有没有。
- 时间词：2026、最近、现在、今年、近一个月。
- 排除词：教程、原理、新闻、官网软文、广告招商、过旧内容。

这是最适合我们短期融合的能力。

### Mail-Hunter 能补联系，但实现要重写

它的做法是用搜索语法找域名相关页面，再用正则提取邮箱。

我们可以借思路，不直接复制代码：

- 对已有官网/主页做 contact/about 页面探测。
- 抽取 email、Telegram、微信、QQ、WhatsApp。
- 每个联系方式标注来源页、可信度、是否人工确认。

## 3. 我们和开源工具的差距

我们已经比很多普通脚本强的地方：

- 已经聚焦“动态住宅 IP”，不是泛 B2B。
- 已经有 2026 时效过滤。
- 已经有线索池、补联系方式、跟进、反馈。
- 已经能区分个人/小团队/工作室，而不是只找公司。
- 已经能解释为什么这条线索值得看。

目前差距最大的是：

1. 数据量还不够稳定  
   用户期望点一次自动采集，就从全部来源拉出较多候选；现在仍容易受来源阻塞、关键词命中少、过滤过严影响。

2. 来源策略还不够“矩阵化”  
   现在有很多源，但更像列表；缺少像 QueryTool 那样按“场景 x 痛点 x 平台 x 时间”系统化生成查询。

3. 联系方式补全还弱  
   开源工具普遍重视 phone/email/website/social。我们现在能给下一步建议，但真实可触达信息还不足。

4. 用户入口还可以更傻瓜  
   成熟工具一般是“输入目标客户 -> 自动采集 -> 看结果 -> 导出/跟进”，而不是让用户理解一堆模块。

5. 任务进度和失败原因要更直观  
   用户点击采集后要马上知道：正在跑哪些平台、成功多少、过滤多少、失败原因是什么、下一次怎么处理。

## 4. 不建议融合的能力

暂时不做：

- 自动私信、自动评论、自动关注。账号风险高，且容易变成灰产工具。
- 复杂 AI Agent。现在没有稳定 API Key，也没有成熟知识库，先不上虚的。
- 过重的 B2B 公司库。用户当前没有公司和发票能力，先找个人、小团队、工作室。
- Google Maps 本地商家大规模采集。和动态住宅 IP 主线弱相关。
- 黑盒付费数据源依赖。可以预留接口，但当前工具要能本地跑、低成本跑。

## 5. 推荐融合路线

### 第一优先级：搜索矩阵引擎

借 QueryTool 思路，把采集从“固定来源列表”升级成：

`客户画像 x 痛点词 x 购买意向词 x 平台 x 时间窗口`

输出一批查询任务，自动分发到 GitHub、Gitee、V2EX、SegmentFault、WeAreSellers、Amazon Seller Forum、知乎、贴吧、微博、B站、小红书、抖音、Bing/Google 可访问结果等来源。

目标：

- 默认全部来源。
- 默认只保留 2026 年及以后。
- 默认排除教程、新闻、广告、同行官网文章和老帖。
- 每条线索必须显示平台、来源、原文时间、发现时间、为什么留下。

### 第二优先级：联系人补全瀑布流

借 Mail-Hunter、OpenOutreach email finder 思路：

1. 先从线索正文找微信/QQ/TG/邮箱/手机号。
2. 再看主页/作者页/个人简介。
3. 再看官网 contact/about 页面。
4. 最后给人工搜索建议。

目标不是 100% 自动找到，而是让用户每条线索知道：

- 已找到什么联系方式。
- 还缺什么。
- 下一步去哪里找。

### 第三优先级：任务进度和失败诊断

借 OpenOutreach 任务队列思路：

- 每个平台一条任务状态。
- 显示运行中、成功、过滤、失败、受阻。
- 失败必须有人话原因：验证码、未登录、403、低质过多、无 2026 内容、来源超时。

### 第四优先级：极简产品入口

借 Bright Data 项目的自然语言入口，但先不用 AI：

- 用户只选“全部目标客户”或某个客户画像。
- 点击“自动找线索”。
- 系统自动跑全部可用来源。
- 结果统一进“线索池”。
- 高级设置收进“工具箱”，日常不暴露。

## 6. 结论

这次下载的开源项目里，没有一个能直接替代 Lead Radar。

最有价值的不是复制代码，而是吸收三类能力：

- OpenOutreach 的流程和任务架构。
- QueryTool 的搜索矩阵思想。
- Mail-Hunter/Google Maps 工具的联系方式抽取、导出和进度反馈。

下一步最值得做的是：实现“搜索矩阵引擎 + 默认全部来源 + 2026 强过滤 + 联系方式补全瀑布流”。这会直接提升线索数量和可用性，比继续加按钮、做复杂 AI、做高级面板更有收益。
