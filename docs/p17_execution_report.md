# P17 执行报告：找线索入口产品化

日期：2026-06-13

## 目标

把“找线索”做成傻瓜式主入口：用户先选择客户类型和来源范围，再点击开始找线索，最后直接看到可判断、可点击、可跟进的线索。

## 已完成

1. 新增找线索服务层 `app/services/lead_finder.py`
   - 支持按客户类型筛选：TikTok 矩阵、亚马逊多账号、Shopee/Lazada/Shein 店群、爬虫/数据采集团队、海外社媒矩阵、独立站/Shopify、账号注册/养号服务商。
   - 支持按来源范围筛选：国内优先、已登录平台、技术社区、跨境卖家、全部来源。
   - 统一输出线索卡片需要的信息：来源、平台、评分、信号、客户类型、原文时间、发现时间、为什么值得看、下一步动作、是否缺联系方式。

2. 重做 `/leads` 页面
   - 页面改为“选客户类型 -> 选来源 -> 开始找线索 -> 看结果”的单一路径。
   - 去掉了原来偏后台的复杂表格感，改成卡片式结果。
   - 每条线索都显示原文时间和发现时间，避免用户误判旧线索。
   - 原文链接、详情链接均可点击。

3. 接入开始找线索动作
   - 新增 `POST /leads/search`。
   - 普通来源调用现有 ingestion 管线。
   - 已登录平台来源调用 session collector。
   - 完成后返回 `/leads` 并显示本次读取、入库和高意向数量。

4. 收紧联系方式判断
   - 主页/原文链接不再算真实联系方式。
   - 只有微信、Telegram、邮箱、网站或明确联系备注才算有联系方式。
   - 这样“待补联系方式”更接近真实销售动作。

## 当前限制

1. P17 解决的是“入口和结果可读性”，还不是完整销售自动化。
2. 已登录平台采集仍然依赖平台状态，遇到验证码、频控、登录态异常会暂停。
3. 当前只展示筛选后的线索，不会自动判断“今天必须联系哪三个人”，这属于 P18/P19。
4. 线索质量仍取决于关键词、来源和平台可读性，后续需要继续优化推荐策略和反馈闭环。

## 下一步

P18：把“待联系”做成真正的补联系方式工作台。重点不是多一个页面，而是让用户知道每条线索缺什么、该去哪里找、补完后进入跟进。

P19：把“今日作战”做成每天只给少量高价值动作，避免用户被大量功能和线索淹没。
